# Memory Hierarchy: From Registers to RAM

## 1. The Registers Gap

Unit 1 ended with a physical observation: the CPU's internal storage — its Registers — is extremely limited. A 64-bit CPU might have 16 to 31 general-purpose registers, each holding 64 bits. That gives the CPU roughly 128 to 248 bytes of fast, directly programmable storage. A real program — even a simple one — works with thousands or millions of data values: variables, arrays, strings, and the program's instructions themselves. Registers alone cannot hold all of this.

This constraint is what led to the development of **Main Memory (RAM)**: a large, external storage where programs and data reside. But RAM introduced a problem. It is dramatically slower than registers. And that single fact — the speed gap between what is fast and what is large — is the reason the entire **memory hierarchy** exists.

---

## 2. The Physical Tradeoff: Speed vs. Size vs. Cost

In hardware design, there is an unavoidable physical constraint:

> **Faster memory is smaller and more expensive. Larger memory is slower and cheaper.**

This is not a design choice. It is a consequence of physics and manufacturing:

- **Fast memory** (like registers) uses more transistors per bit of storage. Each bit in a register is built from multiple transistors arranged in a stable circuit (a **flip-flop**) that holds its value as long as power is supplied. No refreshing needed. This makes access extremely fast, but physically large and expensive per bit.

- **Dense memory** (like RAM) uses fewer transistors per bit. Each bit in typical RAM (**DRAM** — Dynamic RAM) is stored as a tiny electrical charge on a miniature capacitor, using only **one transistor and one capacitor** per bit. This is far more compact, but the charge leaks over time, requiring the hardware to **refresh** every cell thousands of times per second. This refresh overhead, combined with the longer electrical distance to the CPU, makes DRAM significantly slower.

The result: you cannot build a computer with terabytes of register-speed memory. The chip would be physically enormous and prohibitively expensive. Instead, computer architects solved this problem by building a **layered system** — a hierarchy — where each layer makes a different tradeoff between speed, size, and cost.

---

## 3. The Hierarchy

The layers, from fastest to slowest:

```
┌─────────────────────────────┐
│       CPU Registers         │  ← Fastest, smallest (~128-248 bytes)
├─────────────────────────────┤
│        L1 Cache             │  ← Very fast, small (32-64 KB per core)
├─────────────────────────────┤
│        L2 Cache             │  ← Fast, moderate (256 KB - 1 MB per core)
├─────────────────────────────┤
│        L3 Cache             │  ← Moderate, larger (4-64 MB shared)
├─────────────────────────────┤
│     Main Memory (RAM)       │  ← Slow, large (4-128+ GB)
├─────────────────────────────┤
│    Secondary Storage        │  ← Very slow, very large (256 GB - TBs)
│    (SSD, HDD)               │
└─────────────────────────────┘
```

Each level acts as a **buffer** for the level below it. The CPU works directly with registers. When it needs data that is not in a register, it checks the cache. If the data is not in cache, it goes to RAM. If it is not in RAM, it goes to disk. Each step down is a significant increase in latency.

For assembly programming, we are primarily concerned with the top three layers: **Registers**, **Cache**, and **RAM**. Secondary storage (SSDs, hard drives) is managed by the operating system and is not directly relevant to instruction-level programming.

---

## 4. Registers in the Hierarchy

We covered registers in detail in Unit 1. In the hierarchy, their position is defined by three properties:

- **Location:** Physically inside the CPU core, wired directly into the ALU and control unit.
- **Speed:** Accessed in **1 CPU cycle** (sub-nanosecond at modern clock speeds).
- **Size:** Extremely limited — roughly 128-248 bytes of general-purpose storage.

Registers are the only layer of memory that assembly instructions can name directly. When you write `add rax, rbx`, you are telling the CPU to use data that is already sitting in its fastest storage. The performance examples from Unit 1 (register operations at 1 cycle vs. memory-referencing instructions at 4-10+ cycles) demonstrate why keeping data in registers matters.

## 5. Cache: The Invisible Accelerator

### What Cache Is

Between the registers and RAM sits the **cache** — a block of fast memory built directly onto the CPU chip using a technology called **SRAM (Static RAM)**. Unlike DRAM, SRAM stores each bit using a stable circuit of **six transistors**. No refresh cycles are needed, making it significantly faster than DRAM. But six transistors per bit (compared to DRAM's one transistor and one capacitor) means SRAM is physically larger and more expensive per bit. This is why cache is measured in kilobytes or megabytes — the CPU chip physically cannot fit more.

### Why Cache Exists

The gap between register speed (~1 cycle) and RAM speed (~100-300 cycles) is enormous. Without cache, every time the CPU needed data from memory, it would sit idle for hundreds of cycles waiting. Cache fills this gap by keeping **copies of recently and frequently accessed data** close to the CPU, so most memory requests can be served without reaching all the way to RAM.

### Cache Levels

Modern CPUs typically have three levels of cache, each progressively larger and slower:

- **L1 Cache** (Level 1): Closest to the CPU core. Typically split into two halves: one for **data** (L1d) and one for **instructions** (L1i). Size is typically 32-64 KB per half, per core. Latency is approximately 3-5 cycles. Each core has its own private L1 cache.

- **L2 Cache** (Level 2): Larger and slightly slower than L1. Acts as a backup when L1 does not contain the requested data. Size is typically 256 KB to 1 MB per core, with latency around 10-20 cycles. Usually private to each core.

- **L3 Cache** (Level 3): The largest and slowest cache level. Shared among all cores on the chip. Size ranges from 4 to 64 MB total. Latency is approximately 30-70 cycles. Acts as a shared pool so that data needed by multiple cores does not always require a trip to RAM.

### Locality: Why Cache Works

Cache is effective because of a property of real programs called **locality**:

1. **Temporal Locality:** If the CPU accessed a piece of data recently, it is likely to access it again soon. A loop variable, for example, is used every iteration.

2. **Spatial Locality:** If the CPU accessed a memory address, it is likely to access nearby addresses soon. Iterating through an array, for example, touches consecutive locations.

When the CPU requests data from a memory address, the cache hardware checks whether a copy exists in L1. If it does, this is a **cache hit** — the data is returned in a few cycles. If not (a **cache miss**), the request propagates to L2, then L3, and finally to RAM.

When data is fetched from a slower level, the cache hardware does not bring just the requested value. It fetches a block of contiguous memory called a **cache line** (typically 64 bytes on modern CPUs). This is spatial locality in hardware: by loading a neighborhood of data at once, the cache anticipates that the CPU will need nearby values soon.

### Cache and the Assembly Programmer

As established in Unit 1, cache is **managed entirely by hardware**. You do not write assembly instructions to explicitly load data into cache or evict entries from it. The CPU's cache controller handles this transparently.

However, understanding cache behavior still matters because:

- **Sequential memory access** (walking through an array in order) is cache-friendly — spatial locality ensures most accesses are hits.
- **Random memory access** (jumping to scattered addresses) is cache-unfriendly — it causes frequent cache misses, each costing hundreds of cycles.
- The performance difference between cache-friendly and cache-unfriendly code can be **10-100x**, even when the instruction count is identical.

You cannot control the cache directly, but you can write code that works *with* it rather than against it.

## 6. Main Memory (RAM): The External Warehouse

### What RAM Is

**RAM (Random Access Memory)** is the primary external storage for running programs. "Random Access" means the CPU can request data at any address in any order — unlike sequential storage (such as magnetic tape), where data must be accessed in a linear sequence.

Modern RAM is built from DRAM — billions of one-transistor-one-capacitor cells packed onto chips that sit on the motherboard, physically separate from the CPU. The CPU communicates with RAM through a **memory bus**: a set of electrical pathways that carry address signals, data signals, and control signals between the two.

### RAM's Role in Program Execution

When you run a program, the operating system loads it from secondary storage (disk) into RAM. The program's instructions, its data (variables, arrays, strings), and its stack all reside in RAM. The CPU then fetches instructions and data from RAM — through the cache hierarchy — into registers for execution.

From the CPU's perspective, RAM is a flat, linear sequence of addressable locations. Each location holds one **byte** (the smallest addressable unit, as defined in Extras-3 of Unit 1). On a system with a standard 8-bit byte, a machine with 4 GB of RAM has roughly 4 billion individually addressable locations, each identified by a unique numeric **address**.

### RAM Speed

RAM is slow relative to the CPU:

- **Latency:** Approximately 100-300 cycles for a single access (compared to 1 cycle for a register).
- **Throughput:** Modern DDR5 RAM can sustain tens of gigabytes per second of data transfer, but every individual access still carries the latency cost.

This is consistent with what we observed in Unit 1: a memory-referencing instruction like `mov rax, [address]` costs 4-10 cycles when the data happens to be in cache (a cache hit), but can cost **hundreds of cycles** when the data must be fetched from RAM (a cache miss).

The entire cache hierarchy exists to hide this latency. In practice, because of locality, most memory accesses in a well-structured program are served by cache. The CPU only waits for RAM on cache misses.

## 7. Volatility: What Happens When Power Is Lost

Every layer of the memory hierarchy we have discussed — **registers, cache, and RAM** — is **volatile**. This means all data stored in them is lost the instant the computer loses power.

- **Registers:** Flip-flop circuits require continuous electrical current to maintain state.
- **Cache (SRAM):** Same principle — the six-transistor circuits are stable only while powered.
- **RAM (DRAM):** The capacitor charges that represent bits drain almost immediately without power. Even *with* power, they must be refreshed thousands of times per second to prevent data loss.

**Secondary storage** (SSDs, HDDs) is **non-volatile** — data is retained without power. This is why programs and files live on disk and are loaded into RAM only when needed.

This distinction explains a fundamental fact about program execution: the CPU cannot execute instructions directly from an SSD or hard drive. The program must first be copied into volatile RAM, and from there, the cache hierarchy pulls pieces of it into registers for execution. Every time you "open" a program, you are performing this transfer from non-volatile to volatile storage.

## 8. The Numbers

Here is a consolidated view of the hierarchy with typical latency and capacity:

| Layer | Technology | Typical Size | Typical Latency | Volatile? |
|-------|-----------|-------------|----------------|-----------|
| Registers | Flip-flops | ~128-248 bytes | ~1 cycle (<1 ns) | Yes |
| L1 Cache | SRAM | 32-64 KB / core | ~3-5 cycles (~1-2 ns) | Yes |
| L2 Cache | SRAM | 256 KB - 1 MB / core | ~10-20 cycles (~3-7 ns) | Yes |
| L3 Cache | SRAM | 4-64 MB shared | ~30-70 cycles (~10-20 ns) | Yes |
| RAM | DRAM | 4-128+ GB | ~100-300 cycles (~30-100 ns) | Yes |
| SSD | Flash / NAND | 256 GB - TBs | ~10,000+ cycles (~10-100 μs) | No |
| HDD | Magnetic | 1-20+ TB | ~10,000,000+ cycles (~5-10 ms) | No |

Each step down increases capacity by roughly 10-1000x while increasing latency by roughly 3-10x. This is not coincidence — it is a direct consequence of the physics and manufacturing tradeoffs described in Section 2.

---

## 9. Why This Matters for Assembly

In high-level languages, you write `int x = 5;` and the compiler decides where `x` lives — in a register, on the stack (in RAM), or somewhere else. You never think about the memory hierarchy.

In assembly, **you** make these decisions. Every `mov` instruction that loads from or stores to a memory address is a direct interaction with the hierarchy. The cost difference between:

```asm
add rax, rbx          ; Both operands in registers: 1 cycle
```

and:

```asm
add rax, [some_addr]  ; One operand from memory: 4-10+ cycles (cache hit)
                      ;                          100-300+ cycles (cache miss)
```

...is the difference between fast code and slow code. The hierarchy explains *why* that gap exists and *why* assembly conventions (like keeping frequently used values in registers) are not arbitrary rules but direct responses to hardware physics.
